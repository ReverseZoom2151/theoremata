## Focused answers to the four questions

**(a) Is our planning first-class & revisable, or static decomposition?**
**First-class and revisable — this is a genuine strength, not a gap.** `blueprint_generate.rs::plan_and_prove` closes a real generate → drive → **refine** loop: the `BlueprintRefiner` seam re-decomposes *failed* lemmas while preserving proved ones, bounded by `max_rounds`, with monotone coverage and cycle-rejection. `refine_ops.rs` adds an inner/outer scheduler that re-decomposes on stall. The gap is **not** rigidity — it is (i) revision is **only failure-triggered** (no a-priori plan critique) and (ii) **no HITL review** of the plan and **no budget-aware** planning. So: revisable in the "re-plan around a dead end" sense the book praises; missing the "review/approve/allocate-budget" dimension.

**(b) Do we reflect on the PLAN, or only on proof errors?**
**Mostly on proof errors; only structurally on the plan.** `repair.rs` and `refine_ops.rs` reflect on *proof failures*. `critic.rs` is the one component that reflects on plan *structure* (circular deps, unjustified gaps, over-general statements) — but it runs on the proof DAG, and plan revision is driven by `RunReport` proof outcomes, not by a critic scoring a blueprint's plausibility **before** proving spend. **Missing: a plan-critic that vets a freshly generated blueprint (decomposition quality, redundancy, dead-end risk) prior to execution.**

**(c) Meta-tools & dynamic tool selection, or fixed dispatch?**
**Fixed dispatch, no meta-tools.** `worker.py` is a static `if/elif` table keyed on a caller-supplied `tool` name across 94 tools — the model does not choose from a described catalog, and there is no MCP/tool-manifest surface. Routing exists only over *search strategies* (`hybrid_search.rs`), not over worker tools. **No meta-tools**: planning, self-critique, and budget-spending are orchestration code the model executes *inside*, never tools it can *call* (`make_plan`, `critique`, `spend_budget`, `decompose`). This is the single biggest structural divergence from the book.

**(d) Are team/method_transfer/consolidate real agents, or functions?**
**Functions/passes over shared graph state, not autonomous agents.** They are role-flavored transformations sharing the proof-graph DB as a de-facto "shared ontology," but there is **no per-agent message loop, no A2A protocol, no dynamic capability-based delegation.** The one place we *do* realize a book multi-agent pattern well is **Critic-Reviewer** (`critic.rs` + `formal.rs` gate + `PoolMetaGate` vs producer `sketch`/`repair`), which is a true producer/critic separation even though it's wired as passes.

## TOP 3 GAPS to build (ranked)

1. **META-TOOL LAYER (plan / critique / budget / decompose as callable tools).** *Highest leverage.* Today planning, self-critique, and TTC budgeting are code the model runs inside; the book's most modern examples (OpenAI Deep Research exposing plan/search/code steps; ADK `AgentTool`) make these **first-class tool calls**. Expose `make_plan`, `critique_self`, `redecompose`, and `spend_budget` as tools the model invokes and observes — turning `blueprint_generate`, `critic.rs`, `refine_ops`, and `ttc.rs` from fixed control flow into an agent-steerable toolset. Also unblocks **dynamic tool selection** and an **MCP server** wrapping `worker.py`. Builds directly on existing modules; no new theory.

2. **A-PRIORI PLAN-CRITIC + HITL PLAN CHECKPOINT.** Add a critic seam that evaluates a freshly generated blueprint *before* proving spend (decomposition quality, redundancy, dead-end / over-generality risk — `critic.rs` already knows these failure modes) and a checkpoint that can surface the plan for human approval/edit (Deep Research's collaborative plan review). Converts our planning from purely failure-triggered revision into review-then-execute, and adds the missing HITL dimension the book stresses. Reuses `critic.rs` findings taxonomy + `blueprint_generate::PlanConfig`.

3. **FIRST-CLASS AGENT SEAM (roles + A2A messages + agent-as-tool), budget-aware.** Promote `team.rs` / `consolidate.rs` / `method_transfer.rs` from passes to real agents behind a common trait (role, goal, tools) that communicate via an explicit message protocol over the proof-graph "ontology," invocable **agent-as-tool**, and make the planner/refiner **budget-aware** (plan and reallocate compute across blueprint items by expected payoff, feeding `ttc.rs`). This realizes the book's hierarchical/supervisor and (later) debate/consensus models and closes the budget-aware-orchestration gap.

## Verdict: is a unified system-prompt + meta-tool layer warranted?

**Yes for the meta-tool layer — strongly and specifically; qualified yes for a unified system-prompt layer.**

- **Meta-tool layer: warranted and high-value.** This section's throughline (Reflection-as-a-call, ADK `AgentTool`, OpenAI Deep Research exposing plan/critique/search/code as inspectable steps, MCP) is exactly the surface Theoremata lacks. We already own the *hard* parts — a real revisable planner, a durable graph-grounded critic, an inner/outer refinement scheduler, TTC budgeting — but they are locked inside orchestration code instead of being callable, observable tools. Exposing them as meta-tools is the cheapest path to the biggest capability jump and is fully buildable on existing modules (Gap 1).

- **Unified system-prompt layer: warranted but secondary.** The chapters don't argue for it directly (they show *per-role* prompts), but the Producer/Critic and multi-agent material implicitly assumes a shared instruction substrate. Our per-ROLE-only prompts risk drift on cross-cutting invariants (untrusted-text policy, abstention, output contracts). Worth doing, but it is an enabling refactor, not a capability unlock — sequence it *after* the meta-tool layer, and let the meta-tools carry the shared contracts.
