## Core contribution
Aristotle is a hybrid formal/informal theorem prover that produced **machine-verified Lean 4 (Mathlib) solutions to 5 of 6 IMO 2025 problems** = gold-medal-equivalent, concurrent with ByteDance's Seed-Prover. Its architecture is exactly the three-pillar shape Theoremata is building toward: (1) a highly parallel **Monte Carlo Graph Search (MCGS)** over Lean states with one transformer serving as both policy and value; (2) a **lemma-based informal-reasoning outer loop** (informal proof → decompose into lemmas → autoformalize → prove via search → iterate on formal feedback); (3) a dedicated **geometry solver (Yuclid**, a 500×-faster open-source C++ DD/AR engine descended from AlphaGeometry). "Solved" means a complete Lean proof with no `sorryAx`/unsound axioms — verification-first.

## Key techniques / architecture
### 1. Search algorithm (MCGS)
- Ingests a Lean code block and **replaces all `sorry`s with proofs**. Builds on MCTS + learned value, AlphaZero/Expert-Iteration lineage; policy is *generative* (samples Lean tactic strings), same lineage as HyperTree Proof Search, ABEL, DeepSeek-Prover-V1.5, AlphaProof.
- **States** = Lean goals split by goal *up to metavariables*: if an action yields multiple goals none containing metavariables → one state per goal; else keep a single combined state. Goals from an explicit `sorry` placeholder must be split, so they are not allowed to carry metavariables.
- **Actions** = arbitrary Lean code fragments (single tactic, tactic sequence, may include informal comments). An action can produce multiple states → **hypertree** (à la HTPS). Can be viewed as an alternating-level tree (choosing a state = a move in a two-player game).
- **Action history in prompt**: theoretically unnecessary (state machine is memoryless) but empirically reduces circular trajectories and keeps the model on cohesive strategies instead of branching everywhere.
- **Equivalences → graph, not tree.** States equivalent if equal in goal expressions, local-context expressions, and local variable names. Actions equivalent if their state transitions are equivalent. Collapsing equivalents turns the hypertree into a **hypergraph** → Monte Carlo Graph *Search*. Per-step worst case degrades O(D)→O(V) but no CPU bottleneck in practice. (Known `aesop` counterexample: uses global state; worked around.)
- **Search strategy = PUCT with AND/OR minimax.** Prior policy ≈ **empirical distribution of sampled actions** (deliberately *not* sequence logprobs, which bias against actions with multiple equivalent textual forms). **Progressive widening** [Couëtoux] over the huge tactic action space. A **state is proven if ANY action succeeds (OR)**; an **action succeeds only if ALL its resulting states are proven (AND)** → finding a proof = minimax. Search picks the action with highest upper-confidence bound, then targets its **hardest resulting state (lowest lower-confidence bound)** — attacking the bottleneck first.
- **Disproving / pruning:** each single-goal state is **augmented with a transition to the logical negation of the goal**, to which search budget can be allocated → lets the system prune and disprove.
- **Interleaving informal reasoning (3 output channels co-evolved in RL):** (a) hidden chain-of-thought with a **dynamically set thinking budget** before each action; (b) **informal comments** inside actions (give the model visibility into earlier thinking, since the hidden CoT is not later accessible); (c) formal Lean code. Comments are hand-enforced early in training; later the RL signal alone produces ubiquitous comments. The dynamic thinking budget is also a training-time signal for which traces are genuinely useful (only train on useful traces).
- **Postprocessing:** a proof = acyclic subgraph of the search hypergraph with a single source, only successful actions, exactly one action per state, all resulting states per action. After search: apply linter suggestions, skip redundant action sequences, condense/simplify offline (a state may source multiple proofs at different costs).

### 2. Reinforcement learning / data
- **Single model for policy + value.** Expert-iteration style from a pretrained LLM.
- **Statement autoformalization system**: initial autoformalization → judge using Lean-REPL signals → correction (dataset too large to formalize by hand).
- Train policy on **search-found proofs filtered by nontriviality**; train value on proven states + nearby disproven/hard-unproven states. When an informal proof exists, a **faithfulness judge** filters formal proofs unaligned to the informal one.
- **Hindsight Experience Replay**: render proofs of non-root states as if they were the root statement (reuse of every solved subgoal).
- **Test-Time Training (TTT):** at inference, iterate: (1) attempt the main problem + all lemmas from the outer loop; (2) if unsolved, **retrain the model on search traces from those attempts**. Improves efficiency + specialization; cross-pollinates lemmas across different sketches; helps master rarely-seen Mathlib APIs by learning from ad-hoc chains of basic commands.

### 3. Lemma-based reasoning (outer loop) — Fig. 2 pipeline
NL queries on the target theorem: (1) ask for an **informal proof**; (2) ask to **restructure it as a sequence of interdependent lemmas, each with a very short proof**; (3) ask for **Lean formalizations of the lemma statements**; (4) send to Lean REPL, feed error messages back, ask for **corrections**. Each step has sub-queries for self-reflection/editing. Search proves lemmas first (as proven background in the code block), then the target.
- **Iteration with formal feedback:** on failure, annotate lemmas as proved/unproved, then: (1) revise the lemma list keeping proved lemmas + adding new ones; (2) re-formalize; (3) re-correct via REPL. Prompt explicitly instructs the model to consider two failure modes: strategy unworkable (must fundamentally change) vs. strategy sound but not granular enough for search.

### 4. Geometry — Yuclid (open-source, Apache 2.0, C++)
- DD/AR engine, **up to 500× faster than AlphaGeometry-1**. Solves 17/30 AG-30 by pure DD/AR (saturates rest) in **~0.4s on one 3.1 GHz core** with prebuilt numeric diagrams.
- **Numerical rule matcher** first: find rules whose hypotheses+conclusions are numerically true (detect similar triangles, midpoints, bisectors in the diagram), then add all rules about those configurations — avoids redundant symbolic work later.
- **Deduplicating statements** via a dictionary storing current progress toward each statement.
- **AR optimizations:** keep the running echelon form of the linear system, update on each new statement; store each pending statement's form reduced modulo established equations so proving the same statement resumes instead of restarting. Raw-pointer memory management (STL/Boost).
- **Extensions beyond AG-1:** new AR table for **squared lengths** (encodes Pythagoras `AB⊥CD ⟺ AC²+BD²=AD²+BC²`); optional **law of sines** (adds sines to ratios table, solves 1 more AG-30 problem but 10× slower, off by default); **equations-as-statements** (any AR equation wrappable as a Statement, e.g. parallelogram law without new predicates); mined **new rules** from data-gen problems needing auxiliary constructions on ≤4 points.

### Lean interaction (Appendix B)
Stateless **Lean REPL** (based on leanprover-community/repl), hosted on many CPU-only GKE machines, per-request routable — every request carries all info to reconstruct the state. On a found proof: render a **self-contained Lean file**, run through Lean to check kernel errors, **audit which axioms are used**, and check which file lemmas were necessary. Also applies stricter well-typedness checks on tactic-produced proof terms to **eagerly catch errors** before the kernel.

## Results / benchmarks
- **IMO 2025: 5/6 formally verified in Lean** (all but P6, the hardest, also unsolved by other gold-equivalent AIs). Gold-equivalent. Only human step per problem = hand-translating the informal statement into a formal statement (all lemma statements were **autoformalized**). For problems requiring identifying an unknown (e.g. c=4 in P3), candidates first generated by an LLM with math-programming tools.
- Scaling levers used at IMO: model **>200B params**; many parallel lemma-pipeline instances; multiple formal-feedback iterations; TTT.
- Solved-problem highlights: P1 (Sunny Lines) found a *non-human* algebraic derangement proof + explicit counterexample checking up to n=5; P3 (Bonza) invented auxiliary `def S := {p | Prime p ∧ f p > 1}`; P4 repaired a subtly wrong informal proof during formalization; P5 invented invariant `f(k)=k√2/(2k−1)`, later replaced a calculus lemma with `nlinarith`, used the obscure `filter_upwards`.
- Beyond contests: novel Mathlib theorems (Niven's, Gauss-Lucas, char-poly eigenvalues), contributions to PFR and Generalized Quantum Stein's Lemma projects, and validated Terence Tao's Lean *Analysis I* textbook — **found 4 exercises false as written** (gave counterexamples) and found 2 with unnecessary hypotheses.

## Novel vs SOTA-2026
Aristotle *is* the 2026 frontier of formal ATP (co-frontier with Seed-Prover). Genuinely notable: (1) **step-wise MCGS with generative policy reaching gold**, proving whole-proof (Seed-Prover) is not the only path; (2) **empirical-action-distribution prior instead of logprobs** to avoid textual-multiplicity bias; (3) **negation-augmented states** for in-search disproving; (4) **AND/OR minimax bottleneck-first** state selection; (5) **TTT at inference** cross-pollinating lemmas; (6) **Yuclid** as a 500× DD/AR engine with squared-length AR table. The convergent evolution with Seed-Prover (geometry subsystem + informal decomposition into lemmas + RL on Lean-compiler feedback + solved lemmas as context) is called out as the *crucial recipe* for formal gold.

## Adopt-relevance to Theoremata
This paper is the closest published mirror of Theoremata's intended architecture — treat it as a validation + concrete spec of our own roadmap.
- **MCGS driver (direct spec).** We already plan MCTS graph-search; Aristotle gives the exact tricks to adopt: (a) **state/action equivalence to collapse hypertree→hypergraph** (equal goals+context+var-names; actions equal iff transitions equal); (b) **PUCT prior = empirical sampled-action distribution, NOT sequence logprobs** (avoids penalizing tactics with multiple textual spellings — a real, subtle bug we can pre-empt); (c) **AND/OR minimax: OR over actions per state, AND over states per action; pick highest-UCB action then its lowest-LCB (hardest) child** to hit the bottleneck; (d) **progressive widening** over the tactic action space. These are concrete algorithms for our MCGS, not vague guidance.
- **Falsify-before-prove (direct win).** Their **negation-augmented state transition** — give each single-goal state a child that is the goal's logical negation and let search spend budget disproving — is precisely our falsify-before-prove, but *inside* the same search budget. We should wire this into the MCGS node expansion rather than as a separate pre-pass.
- **3+1 gate ≈ their verification.** Their post-search pipeline (render self-contained Lean file → kernel-error check → **axiom audit** → necessary-lemma check → stricter well-typedness on tactic proof terms) maps almost 1:1 onto our gate (compile → axiom/oracle audit → kernel re-check → source scan). Confirms our gate design; the **eager well-typedness check on tactic-produced terms** (catch errors before the kernel) is an add we may not have.
- **Sketch → autoformalize-holes → splice pipeline (direct spec).** Their lemma-based loop *is* our sketch pipeline: informal proof → decompose into short-proof lemmas → **autoformalize lemma statements** → prove holes via search → **iterate on formal feedback** annotating proved/unproved lemmas. Two concrete upgrades for us: (1) the **failure-mode-aware revision prompt** (distinguish "unworkable strategy" vs "not granular enough for search"); (2) the **faithfulness judge** filtering formal proofs that drifted from the informal proof — a quality gate for our splice step.
- **Portfolio / flywheel.** **TTT** (retrain on your own search traces mid-attempt, cross-pollinating lemmas across sketches) is a strong, concrete flywheel mechanism for hard problems; **HER** (re-root every solved subgoal as a standalone training example) multiplies training signal for free. Their **statement-autoformalization system** (autoformalize → REPL-signal judge → correct) is a spec for our autoformalization component.
- **Geometry vertical.** **Yuclid is Apache-2.0 and open source** — we could integrate it directly instead of reimplementing DD/AR, or port its optimizations (numerical rule matcher first; running echelon form; reduce-mod-established resume; **squared-length AR table** for Pythagoras; equations-as-statements). This is the single fastest way to a strong geometry closure engine.
- **Retrieval / library.** "Solved lemmas as context for subsequent reasoning" + TTT cross-pollination is a retrieval-over-proven-lemmas pattern for our library growth.
- **Already-do / not-a-gap flags:** graph search (we plan it), verification-first gate (we have it), portfolio (we have it). The **gaps** they expose: negation-in-search, empirical-distribution prior, faithfulness judge, eager well-typedness check, failure-mode-aware revision prompt, HER re-rooting, TTT.

## Verbatim-worthy details (for implementation)
- **State splitting rule:** split into one state per goal **only when** the action produces multiple goals **and none contain metavariables**; otherwise keep a single combined state. `sorry`-placeholder goals **must** be split → disallow metavariables in them.
- **Equivalence keys:** states — (goal expressions, local-context expressions, local variable names); actions — (resulting state transitions).
- **PUCT prior:** empirical distribution of actions sampled from the generative model (explicitly avoid sequence logprobs). Progressive widening [Couëtoux et al., LION'05].
- **Minimax selection:** state proven ⟺ ∃ successful action (OR); action successful ⟺ all resulting states proven (AND). Explore highest-UCB action, then its lowest-LCB resulting state. Augment each single-goal state with a transition to the goal's logical negation.
- **Lemma pipeline queries (4):** (1) informal proof; (2) restructure as sequence of interdependent lemmas each with very short proof; (3) formalize lemma statements; (4) send to Lean REPL, feed errors back, correct. Each step = multiple self-reflection subqueries.
- **Failure-iteration queries (3):** annotate proved/unproved; (1) revise list keeping proved + add new lemmas (prompt: consider "unworkable strategy" vs "not granular enough"); (2) formalize new lemmas; (3) REPL-correct.
- **Verification pipeline:** render self-contained Lean file → run as command → check kernel errors, audit axioms used, check necessary lemmas; plus REPL stricter well-typedness on tactic proof terms (eager error catching). REPL from leanprover-community/repl; stateless CPU workers on GKE, per-request routing.
- **Yuclid numbers:** 17/30 AG-30 by DD/AR alone, ~0.4s on one 3.1 GHz core (prebuilt numeric diagrams), up to 500× faster than AG-1. Law-of-sines extension: +1 problem but ~10× slower (off by default). Pythagoras via squared-length table: `AB⊥CD ⟺ AC²+BD²=AD²+BC²`.
- **Scaling at IMO:** policy/value model **>200B params**; parallel lemma-pipeline instances; multi-iteration formal feedback; TTT. Only one human step per problem: hand-formalizing the statement.
