Two of these deserve a sentence because a related file makes them look built:

- `components/prover/session/replay.rs` is **not** SafeVerify. Its `GateReplay`
  answers "does this exact tactic sequence still close the root goal", by
  assembling the sequence into a source file and re-running our own gate, to make
  proof minimization safe. SafeVerify replays a compiled Lean *environment*
  constant by constant against a trusted reference to detect a swapped
  declaration. Different input, different trust question, different layer.
- `components/prover/python/theoremata_tools/aristotle_mcp_client.py` is **not**
  an MCP client. Grepping it for `jsonrpc`, `initialize`, `tools/list`,
  `subprocess`, `Popen`, `httpx`, `requests.`, `urllib` returns zero matches. It
  is an offline reference implementation that calls the `aristotlelib` SDK
  directly in live mode. It never speaks the MCP wire protocol. Note that
  `docs/agentic-patterns-mining/A3-memory-learning-mcp-goals-hitl.md:35` claims
  "MCP: consume a server: HAVE" on the strength of this file. That claim is
  wrong and should be corrected.

grant, copying is not permitted at all. These are the ones in this corpus:

`HVM4`, `hvm-bench`, `hvm-compare-perf`, `hvm-core-serialization`, `WanShi`,
`RogersRamanujan-main` (as distinct from `-artifacts-main`, which is MIT),

`proofgrader`, `LeanProgress`, `LeanAgent`, `LeanDojo-v2`, `LeanDojoChatGPT`,
the two harness-resources books, and all thirteen repos in the slice-H batch.

Four of these are load-bearing for code we have already shipped:
`decomposition_admission.rs` and `declaration_lookup.rs` (from
`llm-driven-proof-search`), `infotree.rs` (from Kimina), and
`proof_calibration.py` / `proof_grader.py` / `ewc.py` / `progress_sft.py` /
`lifelong_curriculum.py` (from the five unlicensed Lean-tooling docs). All were
reimplemented rather than copied, which is the correct handling, but the
provenance should be stated in those module headers before any release.

Copyleft and restrictive: `Theorema` GPL-3.0 (ideas only, never transliterate),
`metamath-exe` GPL-2.0-or-later (subprocess invocation only, so no obligation),

product), four Wolfram domain packages GPL 2.0/3.0, `AutoMathText-2.5` under a
restrictive data EULA rather than an OSS licence, and the free Wolfram Engine

## Two disagreements between the docs and the code, recorded deliberately

Neither is a gap; both are places where we knowingly did the opposite of what a
doc asked, and a future reader should not "fix" them by accident.

1. **Comment masking.** `2026-07-new-arrivals` Tier 2 item 23 says the source
   scan must grep comments, because a commented `sorry` tripped HigherDyson's
   validator. Our scanners deliberately mask comments and strings first
   (`lean_soundness.py:24-30`, `axiom_audit.rs:206-207,320` with a test at `:658`
   asserting comments must **not** flag, `formal.rs:835-842`). That is correct
   for false positives, but it makes the false-lemma prose comment in
   `andrews_dhar/thm2_split4` invisible to every layer we have. An
   advisory-severity comment scan sitting outside the gate would recover it
   without weakening the masking. Note `statement_preservation.rs:4137-4162`
   already implements exactly this shape for the TanArctan case, so the pattern
   exists and just is not generalised.

2. **`--allow-incomplete-matches`.** `prover-tooling-arrivals` adopt 2 asked for
   the needle in `fallback_source_scan`, conditional on checking whether `--safe`
   already rejects it. The check was done and is documented in code at
   `external.rs:1310-1317`: it does, so the needle was deliberately omitted and
   the scan was generalised into a shared alias-expanded escape-hatch table
   instead, which is broader than what the doc asked for.
